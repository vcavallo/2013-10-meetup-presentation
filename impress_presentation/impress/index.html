TESTING!

- Always important. More so in this context.

- New reasons to test...

1. Allows potential users to verify that your code is not borked.

2. Facilitates easy contribution (we'll come back to this soon).

3. Easily get a sense of the API ("it should make you a pizza. it should ask what toppings you want.") thanks to RSpec's nice readable format. (we'll come back to this even sooner)

Added bonus: Use Simplecov!

Seeing 100% coverage on passing tests and knowing you're shipping a reliable gem is just a nice feeling...

Documentation. Do it.

- A user should know what your gem does. That doesn't need much explanation, right?

- You are happy when software you are using has documentation. There's a sort of golden rule thing happening here.

Room to GROW

Start documentation small so that when your project is huge, so too shall be your docs.

Documenting helps you write code! Keep your psuedocode around, update it with some extra detail and a few bits of YARD formatting.

There. You now have documentation.

Contributors and Open Source

This seems like a good place to mention licenses.

The MIT license can by used by anyone - make sure to include any current and future contibutors.

Make it clear to any viewers of your code that you are open to contributions.

Give them some general instructions.

Again, tests are going to be important here. Yours and theirs.

(remember that test snippet from earlier about allowing the message to be optional? this one)

Tests - yours and your collaborators - make open source contributions easier and more fun.

We all appreciate the Ruby community - and the fact that anyone can create and publish a gem in minutes is an awesome part of it.

Communities don't have characteristics other than those that belong to its members.

Consistency and predictable behavior are important parts of any programming community.

Since each of us is a small block in the Ruby community, and that community is nothing more than the combination of our joint participation, we should each make an effort to adhere as much as we can to best practices.

In order to keep Ruby awesome!
